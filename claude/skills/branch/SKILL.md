現在のブランチが作業内容に対して適切かを判断し、必要に応じてブランチの作成・切り替えを提案する。

## 手順

### 1. 現状の確認

以下を並列で実行して現状を把握する：

- `git branch --show-current`（現在のブランチ名）
- `git status`（未追跡ファイルを確認。`-uall` フラグは使用しない）
- `git diff HEAD`（変更内容）
- `git log --oneline -5`（直近のコミット履歴）

### 2. ブランチの適切さを判断

以下の基準でブランチの状態を評価する：

**明らかに不適切（新ブランチ作成を提案）：**

- main または master ブランチにいる
- ブランチ名と変更内容が明らかに一致しない
  - 例：`fix/login-bug` にいるのに、新機能追加の変更がある
  - 例：`feat/add-payment` にいるのに、全く関係ない設定ファイルのみ変更している

**適切（そのまま続行）：**

- すでに feature / fix / chore などのブランチにいて、ブランチ名と変更内容が概ね一致している

**判断が難しい（ユーザーに確認）：**

- ブランチ名が汎用的すぎて判断できない（例：`dev`、`work`、`temp`）
- 複数の異なる種類の変更が混在している
- ブランチ名と変更内容が部分的に一致している

### 3. 判断結果に応じた対応

#### ブランチが適切な場合

```
現在のブランチ: <branch-name>
変更内容との整合性: 問題なし

そのまま作業を続けてください。
```

スキルを終了する。

#### 新ブランチの作成が必要な場合

変更内容を分析してブランチ名を提案する：

- 形式：`<type>/<name>`（type は commit と同じ規則に従う）
- name は変更内容を端的に表す英語の kebab-case
- 例：`feat/add-login-feature`、`fix/null-pointer-error`、`chore/update-dependencies`

以下の形式で提示する：

```
現在のブランチ: <current-branch>（<理由：mainブランチ / ブランチ名と変更内容が不一致>）

新しいブランチを作成することを推奨します。

提案ブランチ名: <proposed-branch-name>

このブランチ名で作成しますか？（別の名前を入力することもできます）
```

ユーザーが承認または別名を入力したら、手順4へ進む。

#### ユーザーに確認が必要な場合

```
現在のブランチ: <branch-name>

以下の理由により、このブランチが作業内容に適しているか判断できませんでした：
<理由>

このブランチのまま続行しますか？それとも新しいブランチを作成しますか？
```

ユーザーが新ブランチ作成を選んだ場合は、ブランチ名を提案して手順4へ進む。
そのまま続行を選んだ場合はスキルを終了する。

### 4. ブランチの作成・切り替え

ユーザーが承認したブランチ名（または入力した名前）で操作する：

**新規ブランチを作成して切り替える場合：**

```bash
git switch -c <branch-name>
```

**既存ブランチに切り替える場合**（例：main にいるが対応ブランチが既に存在する場合、または別の feature ブランチに移動する場合）：

```bash
git switch <branch-name>
```

切り替え後に `git branch --show-current` で現在のブランチを確認し、ユーザーに伝える。

## 注意事項

- ブランチの切り替えには `git switch` を使用する（`git checkout` は使わない）
- 未コミットの変更がある状態でブランチを切り替えると変更が引き継がれることをユーザーに伝える
- 強制切り替え（`--force`）はユーザーが明示的に求めた場合のみ行う
